  -----------------------------------------------------------------------
  Core Doctrine: Build one machine. Ship seven toolheads. The chassis is
  the platform. The toolhead is the purpose. The interface between them
  is sacred and universal.

  -----------------------------------------------------------------------

  -----------------------------------------------------------------------
  One-Sentence Summary: The Universal Skateboard Chassis is a single
  standardized 4-wheel drive base carrying power, compute, mobility,
  localization, and diagnostics — onto which any of the 7 robot toolheads
  (Survey, Excavator, Fabricator, Verification, Assembly, Sealer,
  Logistics) snap via one mechanical-electrical-data interface connector.

  -----------------------------------------------------------------------

1. Purpose

The Skateboard exists because building 7 different robots is a supply
chain nightmare that kills exponential scaling.

If every robot type has its own wheels, its own batteries, its own motor
drivers, its own compute board, and its own frame — then at 10 clusters
you're managing 7 different BOMs. At 1,000 clusters you're managing
7,000 different part inventories across continents.

The Skateboard collapses this into one BOM.

  -----------------------------------------------------------------------
  One chassis. One battery. One motor driver. One compute board. One
  wheel assembly. One assembly procedure. Seven swappable toolheads.

  -----------------------------------------------------------------------

2. Strategic Multipliers

Supply Chain Compression

- One SKU for the drive base, ordered in massive volume from one
  supplier

- Volume discount on identical components across every cluster

- Predictable lead times — no custom parts per robot type

- Simple quality control — one incoming inspection procedure

Training Compression

- Local team learns one machine deeply instead of seven superficially

- A resident who maintains skateboards for three months becomes expert
  at the one system that underlies every robot in every cluster

- That expertise transfers instantly to every robot everywhere

Maintenance Compression

- Every spare wheel, battery, motor driver, and compute board is
  interchangeable across the entire fleet

- No "we need the Fabricator-specific motor" delays

- Broken skateboard swapped whole; repaired at central hub

Capability Extension

- New toolhead = new robot capability on existing fleet

- Glass production attachment, furniture fabricator, agricultural planter
  — all work on existing skateboards immediately

- Network capability expands without hardware replacement

- You don't upgrade the robot. You ship a new toolhead.

3. Mechanical Architecture

3.1 Frame

- Rectangular steel or aluminum chassis plate

- Dimensions: approximately 800mm × 600mm × 150mm (tuned to carry
  heaviest toolhead — Fabricator press)

- Four mounting points for wheel/drive modules at corners

- Center-top flat mounting surface for toolhead interface plate

- Lifting handles or forklift slots on both ends

- Frame rated for minimum 500kg combined payload (toolhead + work
  material)

  -----------------------------------------------------------------------
  Design rule: The frame is a table. Flat top. Wheels underneath.
  Toolhead sits on top and locks down. Nothing clever.

  -----------------------------------------------------------------------

3.2 Drive System

- 4 independently driven wheels (skid-steer or Mecanum for
  omnidirectional movement)

- All 4 wheel assemblies are identical and interchangeable

- Each wheel module: brushless DC motor + planetary gearbox + wheel +
  mounting bracket + encoder

- Wheel modules attach to frame via 4-bolt pattern — tool-free quick
  release if using quarter-turn locks

- Designed for outdoor terrain: pneumatic or solid rubber tires, minimum
  200mm diameter

- Speed: 0.5–2.0 m/s operational (not racing — precision matters more
  than speed)

- Turning: zero-radius turn capability required for tight site
  maneuvering

3.3 Battery System

- Single standardized battery pack per skateboard

- Chemistry: LiFePO4 (lithium iron phosphate) — chosen for safety,
  cycle life, temperature tolerance in field conditions

- Capacity: sized for 2–4 hours of continuous operation under heaviest
  toolhead (Excavator or Fabricator)

- Voltage: 48V nominal system voltage

- Battery pack is a self-contained module with BMS (battery management
  system) integrated

- Pack mounts into chassis via slide-rail + latch — tool-free swap in
  under 30 seconds

- Charging contacts on chassis rear — recessed DC contact pads that
  mate with Parasitic Power walk-in dock capture geometry (see:
  Autonomous Charging Dock + Final-Pose Truth Stack blueprint)

- Battery health telemetry: voltage, current, temperature, state of
  charge, cycle count — all reported to House Brain

  -----------------------------------------------------------------------
  The battery is not bolted in. It slides in like a magazine. A resident
  swaps a depleted battery for a charged one and the robot continues
  working in 30 seconds. Zero downtime for charging if spare packs are
  available.

  -----------------------------------------------------------------------

3.4 Compute Module

- Single-board computer: ESP32-S3 or Raspberry Pi CM4 class

- WiFi + Bluetooth for House Brain communication

- USB/UART/SPI interfaces for toolhead data bus

- Runs the local robot controller firmware

- Receives commands from House Brain via API

- Emits telemetry per contract Layer 2

- Stores no persistent state — House Brain database is truth

- Compute module mounts in a sealed enclosure on the chassis

- Enclosure is IP65 minimum — dust and water resistant

- Compute module is a Red/Green swappable module (see Diagnostic
  Protocol)

3.5 Motor Driver Board

- Dual or quad motor driver handling all 4 wheel motors

- 48V input from battery system

- Current sensing per motor channel — feeds House Brain telemetry and
  diagnostic LED system

- Over-current, over-temperature, and stall protection built in

- Motor driver is a Red/Green swappable module

3.6 Sensor Suite (Chassis-Level)

Every skateboard carries these sensors regardless of toolhead:

- IMU (accelerometer + gyroscope) — orientation, tilt, vibration
  signature

- UWB anchor receivers — site-level localization (x, y position)

- Wheel encoders — odometry backup

- Battery state sensors — voltage, current, temperature, SoC

- Ambient temperature sensor — environmental baseline

- Chassis vibration sensor — anomaly detection for drive system health

- GPS module (optional, for inter-site navigation only)

  -----------------------------------------------------------------------
  Toolhead-specific sensors live on the toolhead, not the chassis. The
  chassis knows where it is, how it's moving, and whether it's healthy.
  The toolhead knows what it's doing.

  -----------------------------------------------------------------------

4. Universal Toolhead Interface (UTI)

This is the most critical specification in the entire Skateboard design.
Every toolhead connects to the chassis through one standardized
interface. No exceptions. No per-type variants.

4.1 Mechanical Interface — Self-Seating Mount Architecture

  -----------------------------------------------------------------------
  Mount doctrine: The mount is NOT the structural hero. The toolhead
  self-grounds its active work loads into the soil. The mount handles
  transport, alignment, and retention only. See: Toolhead Mount + Load
  Path Validation Stack blueprint for the full transmutation chain.
  See: Tool-Free Retention + Truth-of-Lock Stack blueprint for the
  retention verification doctrine.

  -----------------------------------------------------------------------

  Load Path Separation:

  - TRANSPORT loads (dead weight of toolhead during mobility): carried
    by mount → chassis → wheels → ground

  - ACTIVE WORK loads (compression, vibration, torque during operation):
    carried by toolhead → toolhead anchor feet → soil. Bypasses chassis
    entirely.

  - The mount never sees full active work reaction forces. It is a
    carriage interface, alignment interface, and retention interface —
    not a structural spine.

  Mount Geometry:

  - Toolhead mounting surface: 400mm × 400mm centered on chassis top

  - Self-seating taper geometry: mount faces are angled so that
    transport loads (gravity, vibration, deceleration) push the toolhead
    INTO the seat rather than loosening it. Wolff's Law principle —
    stress tightens the joint, not loosens it.

  - Keyed profile: toolhead base and chassis cradle share a unique
    asymmetric outline (like a phone case) that prevents incorrect
    orientation and prevents random work impacts from imitating the
    release path.

  Interface Protection:

  - Normally closed: the precision engagement surfaces are covered /
    sealed when no toolhead is mounted. Dirt, dust, and rain do not
    attack the critical geometry while idle.

  - Interface opens only when correct keyed toolhead body is presented
    and seated.

  Release Mechanism:

  - Recessed pressure trigger: release is NOT an exposed button or
    external lever. It is a recessed internal actuation point accessible
    only through the correct keyed access path.

  - Activation requires: correct geometry alignment + insertion into
    recessed access + force threshold. Random vibration, impacts, and
    debris cannot accidentally trigger release.

  - Force-threshold based, not precision-choreography based — works
    even with moderate dust or wear present.

  Retention Verification (Truth-of-Lock):

  - Once toolhead is seated, the mount POSITIVELY LOCKS — not by
    friction or preload alone, but by a physical lock state.

  - Release requires DELIBERATE CHASSIS COMMAND — vibration, thermal
    cycling, dust, and impacts cannot cause passive back-out.

  - Unlock authority is hardened by two-step verification: unlock
    intent signal + physical release action must both occur.

  - True lock state is verified by physical reality, not software
    belief. A mounted toolhead is not trusted as "locked" unless:
    1. Keyed geometry alignment confirmed
    2. Recessed access path reached
    3. Internal pressure threshold crossed
    4. Pressure maintained for dwell time

  - If any verification step fails, the chassis reports mount_state
    as UNCERTAIN to House Brain and does not proceed with operations.

  - See: Tool_Free_Retention_Truth_of_Lock_Stackv1 for the full
    transmutation chain behind the retention verification doctrine.

  Sacrificial Wear Insert:

  - The keyed access geometry and precision seating surfaces are
    contained in a REPLACEABLE INSERT MODULE — not machined into the
    chassis frame itself.

  - Insert has a defined TTL (service life in mount/unmount cycles and
    operating hours).

  - Insert is a Red/Green diagnostic module: Green (within service
    life), Amber (approaching replacement threshold), Red (replace now).

  - Replacement: slide out old insert, slide in new insert. Field-
    serviceable. Tracked in House Brain inventory like any other module.

  Load Rating:

  - Transport dead load: rated for 150kg static (covers Fabricator,
    heaviest toolhead at ~120kg, with margin)

  - Transport dynamic load: rated for 100kg equivalent dynamic
    (acceleration, braking, rough terrain vibration during mobility)

  - Active work loads: NOT rated through the mount. Active work loads
    are the toolhead's responsibility via its own ground anchor / feet.

  - If mount sees active work loads, the toolhead's self-grounding
    system has failed — that is a Red diagnostic condition.

  Swap Time:

  - Toolhead removal: 60 seconds maximum by one person for standard
    toolheads. Fabricator may require two people due to mass.

  - No tools required for swap. No screwdriver, no wrench, no laptop.

  -----------------------------------------------------------------------
  If you need a screwdriver to change a toolhead, the interface is wrong.
  If the mount is absorbing press reaction loads, the toolhead design is
  wrong. If the wear insert needs replacing, that is planned maintenance
  — not failure.

  -----------------------------------------------------------------------

4.2 Electrical Interface

- Single multi-pin connector (industrial grade, keyed, IP65)

- Power bus: 48V DC, 30A continuous from chassis battery to toolhead

- 12V auxiliary: 5A for sensors, LEDs, low-power actuators

- 5V logic: 2A for microcontrollers, sensors

- Motor control channels: 4 configurable H-bridge outputs for toolhead
  actuators (shared from motor driver expansion)

- Analog inputs: 8 channels, 12-bit ADC — load cells, potentiometers,
  temperature sensors

- Digital I/O: 8 configurable GPIO — limit switches, encoders,
  solenoids

- Emergency stop: dedicated pin — toolhead can trigger chassis e-stop,
  chassis can trigger toolhead e-stop

  -----------------------------------------------------------------------
  One connector. Power, data, control, safety. Plug it in and the
  toolhead is live. Unplug it and the toolhead is inert.

  -----------------------------------------------------------------------

4.3 Data Interface

- UART serial bus: primary communication between chassis compute and
  toolhead controller

- Protocol: JSON-over-serial at 115200 baud (matches existing House
  Brain telemetry format)

- Toolhead identifies itself on connection via handshake message:
  { "toolhead": "excavator", "version": "1.0", "serial": "EXC-0042" }

- Chassis compute routes toolhead telemetry to House Brain with
  chassis_id and toolhead_id metadata appended

- Toolhead firmware update: triggered via House Brain command, delivered
  over UART from chassis compute

  -----------------------------------------------------------------------
  The chassis doesn't know what the toolhead does. It knows what the
  toolhead IS. The toolhead announces itself. The House Brain routes
  commands accordingly.

  -----------------------------------------------------------------------

4.4 Diagnostic Interface

- Addressable RGB LED strip connector: dedicated 5V data line to
  toolhead LED array

- Chassis diagnostic LEDs and toolhead diagnostic LEDs are on the same
  addressable bus but different address ranges

- Chassis LEDs: addresses 0–15 (wheels, battery, compute, motor driver)

- Toolhead LEDs: addresses 16–47 (toolhead-specific module diagnostics)

- See Red/Green Diagnostic Protocol blueprint for full LED state machine

6. Version Control Protocol

The Skateboard design will improve over time. This must not fragment the
fleet into incompatible hardware generations.

6.1 Versioning Scheme

  SKATEBOARD v[MAJOR].[MINOR]

  - MINOR version change: backwards compatible with all existing
    toolheads. Better motors, larger battery, improved sensors —
    same UTI connector. Example: v1.0 → v1.1 → v1.2

  - MAJOR version change: UTI connector modified. Existing toolheads
    require an adapter plate to function. Example: v1.x → v2.0

6.2 Compatibility Rules

  - All v1.x skateboards work with all v1.x toolheads. No exceptions.

  - A v2.0 skateboard MUST ship with a v1-to-v2 adapter plate so
    existing toolheads continue to function.

  - Toolhead versions are independent of skateboard versions. A
    toolhead v2.0 Excavator works on a v1.0 skateboard if power and
    data requirements are within v1.0 UTI limits.

  - Maximum adapter chain: one. You cannot stack v1-to-v2 + v2-to-v3
    adapters. At that point, toolhead is redesigned for current UTI.

6.3 Fleet Inventory Tracking

  - Every chassis has a unique serial: SKT-[BATCH]-[SEQ]
    Example: SKT-2026A-0001

  - Every toolhead has a unique serial: [TYPE]-[BATCH]-[SEQ]
    Example: EXC-2026A-0042

  - House Brain tracks chassis ↔ toolhead pairings in robot registry

  - Toolhead swap events are logged with timestamp, chassis_id,
    old_toolhead_id, new_toolhead_id

7. Assembly Procedure

One procedure. For every robot.

7.1 Skateboard Assembly (from shipped components)

  1. Lay frame plate on flat surface
  2. Snap in 4 wheel modules (quarter-turn lock each)
  3. Slide battery pack into rail slot, latch closed
  4. Seat compute module in sealed enclosure, close clips
  5. Seat motor driver board in enclosure, connect harness
  6. Connect sensor harness (single multi-pin connector)
  7. Power on — chassis self-tests, LEDs go through startup sequence
  8. Chassis reports to House Brain via WiFi: "SKT-XXXX online, no
     toolhead attached"

  Time: ~15 minutes by one trained person. No tools required.

7.2 Toolhead Mounting

  1. Position toolhead over chassis mounting surface — keyed profile
     guides alignment (only one orientation fits)
  2. Lower toolhead into self-seating taper — toolhead weight settles
     it into contact with seating faces
  3. Confirm seated: toolhead profile engages cradle, retention
     activates (audible/tactile confirmation)
  4. Connect UTI multi-pin connector (single plug)
  5. Toolhead powers up, sends handshake to chassis compute
  6. Chassis reports to House Brain: "SKT-XXXX now carrying EXC-0042"

  Time: ~60 seconds. No tools required.

7.3 Toolhead Swap (in field)

  1. Chassis parks and powers down toolhead via command
  2. Disconnect UTI connector
  3. Access recessed release trigger through keyed path
  4. Apply pressure to release — retention disengages
  5. Lift off old toolhead (may require two people for Fabricator)
  6. Set new toolhead into cradle, keyed profile seats it, connect UTI
  7. New toolhead handshake — chassis reports new pairing to House Brain

  Time: ~2 minutes. No tools required.

  -----------------------------------------------------------------------
  A cluster of 7 skateboards with 7 toolheads can be reconfigured to
  any combination of roles in under 15 minutes. Need more Excavators
  during site prep? Move toolheads. Need more Assembly during the wall
  phase? Move toolheads. The fleet adapts to the build phase without
  buying more robots.

  -----------------------------------------------------------------------
